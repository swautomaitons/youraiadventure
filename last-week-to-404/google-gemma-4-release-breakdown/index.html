Bug Survival Guide v1.0 Apr 2026

The Complete Gemma 4 Bug Survival Guide

What's broken, what works, and why 2 million people downloaded it anyway. A comprehensive reference covering tool calling failures, vLLM throughput problems, quantization gibberish, and the infinite token loop.

TL;DR

Google released Gemma 4 with broken tool calling, vLLM running at 9 tokens/sec (vs 35 tok/s for competitors), quantized versions producing gibberish, and an infinite token loop bug. All of these have workarounds. Most will get fixed. Despite this absolute chaos, the model is genuinely excellent when it works. This guide is your survival manual.

01 The Bugs

What's broken and how broken it is
Broken Tool Calling Fixed
Status
Shipped broken on April 9, patched April 7 (yes, pre-release fix)
What Happened
Google released Gemma 4 with non-functional tool calling — the headline feature of the entire release. Function calls returned malformed JSON or nothing at all.
Fix Timeline
Community flagged it immediately. Google shipped a patch within 5 days. Works now.
Tip: If you downloaded Gemma 4 before April 7, re-download the model weights. The patched version is a different checkpoint.
vLLM Throughput Open Bug
Status
vLLM runs Gemma 4 at 9 tokens/sec on RTX 4090 (vs 35 tok/s for Qwen 3.5)
Root Cause
vLLM hasn't been optimized for Gemma 4's architecture yet. This is not a model problem — it's an inference engine problem.
Workaround
Use Ollama instead. Ollama runs Gemma 4 at acceptable speeds. vLLM patches expected within 2-3 weeks.
Warning: If your production stack depends on vLLM, you can't use Gemma 4 right now. Period.
GGUF Quantization Open Bug
Status
Quantized versions produce outputs that diverge significantly from full-precision model
What Happened
GGUF quantizations generate gibberish. Not "slightly degraded quality." Gibberish. Community reports show outputs that bear only a passing resemblance to expected behavior.
Fix Timeline
"Whenever someone gets to it." No official timeline. Community-driven fix.
Warning: Skip all quantized versions entirely until the community produces validated quants. Stick to full-precision fp16 for now.
<unused24> Token Loop Open Bug
Status
Under certain prompt conditions, Gemma 4 enters an infinite generation loop producing <unused24> tokens
Root Cause
Not well-documented. Nobody fully understands the trigger conditions yet. Appears to be related to specific prompt patterns.
Workaround
Set stop sequences: ["<unused24>", "<|endoftext|>"]. This will halt generation when the loop starts.
Tip: Always set stop sequences when using Gemma 4 in production. This is non-negotiable until the model update ships.

Proven fixes from the community trenches

Why:Ollama works. vLLM doesn't. It's that simple.
Stop Sequences Essential
# Always set these stop sequences { "stop": [ "<unused24>", "<|endoftext|>", "<|im_end|>" ] }
Purpose:Guard against infinite token loop
Impact:Zero performance cost, prevents catastrophic failures

Tool Calling (Post-Patch) Fixed
Verify Version
Check model checkpoint date. Must be April 7, 2026 or later.
Re-download
If you downloaded before April 7, delete and re-download. The patch is a different checkpoint, not a hot-fix.
Tip: After re-downloading, run a simple tool call test to verify function calling works before deploying to production.
Prompt Engineering Mitigation
  • Keep prompts clear and structured: Ambiguous prompts increase token loop risk
  • Use delimiters: XML tags or triple quotes to separate instructions from content
  • Set max_tokens: Don't let generation run indefinitely even with stop sequences
  • Test edge cases: Empty inputs, very long inputs, special characters all increase bug risk

    05 Production Deployment Checklist

    Before you deploy Gemma 4 to production
    Pre-Deployment Checklist Mandatory
  • Verify model checkpoint date: Must be April 7, 2026 or later
  • Use fp16 weights: Avoid all quantized versions for now
  • Deploy with Ollama: vLLM not ready yet
  • Set stop sequences: ["<unused24>", "<|endoftext|>"] minimum
  • Set max_tokens: Hard cap to prevent runaway generation
  • Test tool calling: Run a simple function call to verify it works
  • Load test: Simulate production load to verify throughput
  • Monitor outputs: Watch for gibberish or token loops in early production
    • Monitoring Essential
    Watch For
    Token loops, gibberish outputs, unusually long generation times, stop sequence triggers
    Alert On
    Stop sequence hit rate > 1%, generation time > 10s for short prompts
    Rollback Plan Safety Net
    Keep Previous Model
    Maintain your previous model (Llama, Qwen, etc.) ready to deploy if Gemma 4 fails in production
    Trigger
    If token loop rate > 5% or output quality degrades, rollback immediately

    Key Takeaways

    • Tool calling shipped broken and got fixed in 5 days. Google released Gemma 4 on April 9 with non-functional tool calling — the headline feature. Patched by April 7. Works now. Re-download if you got it early.
    • vLLM runs at 9 tokens per second on a $1,600 GPU. Qwen 3.5 runs at 35 tok/s on the same hardware. Use Ollama until vLLM optimizations land (2-3 weeks).
    • Quantized versions produce gibberish. GGUF quantizations do not faithfully represent the full-precision model. Skip all quantized versions until validated quants emerge.
    • The model spams <unused24> tokens into infinity. Under certain prompt conditions, Gemma 4 enters an infinite generation loop. Set stop sequences: ["<unused24>", "<|endoftext|>"].
    • Despite all of this, the model is genuinely excellent. 86.4% on agentic benchmarks (13x over Gemma 3). 89.2% on math. Apache 2.0 license. When it works, it's frontier-quality open-source AI.
    • The price is free. The cost is patience. All of these problems will get fixed in weeks, not months. Early adopters absorb the debugging. The reward is a frontier-quality model running locally with zero licensing fees.
