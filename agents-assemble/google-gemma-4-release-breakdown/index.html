Step 1 / 6 · ~3 min

Install Ollama

Ollama is the local LLM runtime that makes running Gemma 4 as easy as pulling a Docker image. It handles model downloads, GPU acceleration, and the inference server — so you can focus on building your agent.

macOS Installation

terminal 
brew install ollama

Or download the official installer from ollama.com/download.

Linux Installation

terminal 
curl -fsSL https://ollama.com/install.sh | sh

Windows Installation

Download the Windows installer from ollama.com/download and run the .exe file.

Verify Installation

terminal 
ollama --version

Ollama automatically starts a local API server on localhost:11434. No configuration needed.

Step 2 / 6 · ~2 min

Pull Gemma 4 Model

Now that Ollama is installed, pulling Gemma 4 is a single command. The model download is about 5GB, so this might take a few minutes depending on your connection.

terminal 
ollama pull gemma:4

Once the download completes, test that the model is working:

terminal 
ollama run gemma:4 "What is an AI agent?"

You should see a response explaining AI agents. If this works, Gemma 4 is installed and ready to accept tool calls.

The gemma:4 tag pulls the latest stable version. For GPU acceleration, Ollama automatically detects and uses your GPU if available.

Step 3 / 6 · ~8 min

Define Your Tool Schemas

This is where Gemma 4's native tool calling shines. You define tools as JSON schemas, and the model understands when and how to call them — without elaborate prompt engineering.

Create a simple tool schema file

tools.json 
{
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "get_weather",
        "description": "Get current weather for a city",
        "parameters": {
          "type": "object",
          "properties": {
            "location": {
              "type": "string",
              "description": "City name, e.g. 'London'"
            }
          },
          "required": ["location"]
        }
      }
    },
    {
      "type": "function",
      "function": {
        "name": "calculate",
        "description": "Evaluate a math expression",
        "parameters": {
          "type": "object",
          "properties": {
            "expression": {
              "type": "string",
              "description": "Math expression, e.g. '42 * 17'"
            }
          },
          "required": ["expression"]
        }
      }
    }
  ]
}

This schema tells Gemma 4 that it has access to two functions: get_weather and calculate. The model will read these schemas and call the appropriate function when the user's query requires it.

Write clear descriptions for each function. The model relies on these to decide when to call a tool. Good descriptions = better function calling accuracy.

Step 4 / 6 · ~5 min

Test Your First Function Call

Now let's test whether Gemma 4 can call your tools. Use the Ollama API to send a prompt that requires a tool call.

test.py 
import requests
import json

url = "http://localhost:11434/api/chat"

with open("tools.json") as f:
    tools = json.load(f)["tools"]

payload = {
    "model": "gemma:4",
    "messages": [
        {"role": "user", "content": "What's 1547 * 23?"}
    ],
    "tools": tools,
    "stream": False
}

response = requests.post(url, json=payload)
print(json.dumps(response.json(), indent=2))

Run the test:

terminal 
python test.py

You should see Gemma 4 return a tool_calls object with the function name calculate and the argument {"expression": "1547 * 23"}.

If you see JSON parsing errors, you're likely using an older version of Gemma 4 from before the April 7 fix. Run ollama pull gemma:4 again to get the latest version.

Step 5 / 6 · ~7 min

Build a Production Agent

Now let's wire up the orchestration loop that executes tool calls and feeds results back to the model. This is the core pattern for any AI agent.

agent.py 
import requests
import json

def get_weather(location):
    # Mock implementation
    return f"Weather in {location}: 22°C, Sunny"

def calculate(expression):
    try:
        return str(eval(expression))
    except:
        return "Error: invalid expression"

TOOL_FUNCTIONS = {
    "get_weather": get_weather,
    "calculate": calculate
}

def run_agent(user_input):
    with open("tools.json") as f:
        tools = json.load(f)["tools"]

    messages = [{"role": "user", "content": user_input}]

    for _ in range(10):  # Max 10 tool rounds
        response = requests.post(
            "http://localhost:11434/api/chat",
            json={
                "model": "gemma:4",
                "messages": messages,
                "tools": tools,
                "stream": False
            }
        ).json()

        assistant_msg = response["message"]
        messages.append(assistant_msg)

        # If no tool calls, we're done
        if not assistant_msg.get("tool_calls"):
            return assistant_msg["content"]

        # Execute each tool call
        for tc in assistant_msg["tool_calls"]:
            fn_name = tc["function"]["name"]
            fn_args = tc["function"]["arguments"]

            fn = TOOL_FUNCTIONS.get(fn_name)
            result = fn(**fn_args) if fn else f"Error: unknown tool '{fn_name}'"

            messages.append({
                "role": "tool",
                "content": result,
                "tool_call_id": tc["id"]
            })

    return "Agent hit max tool rounds"

# Test it
print(run_agent("What's the weather in Tokyo and calculate 42 * 17"))

The 10-round limit prevents infinite loops. Most agents finish in 1-3 rounds. If you're hitting the limit, check your tool descriptions for ambiguity.

Step 6 / 6 · ~5 min

Production Deployment Tips

You've got a working local AI agent. Here's what you need to know before deploying this in production.

1. Error Handling

Always wrap tool executions in try/catch blocks. If a tool fails, return a clear error message so the model can try an alternative approach.

2. Rate Limiting

Local models don't have API rate limits, but you should still cap concurrent requests to prevent resource exhaustion. Use a simple queue or semaphore.

3. Memory Management

Gemma 4 uses about 5GB of RAM when loaded. On production servers, monitor memory usage and restart Ollama if it grows unbounded.

4. GPU Acceleration

If you have a GPU, Ollama will automatically use it. This makes inference 5-10x faster. No configuration needed.

Check GPU usage 
nvidia-smi  # For NVIDIA GPUs
rocm-smi    # For AMD GPUs
