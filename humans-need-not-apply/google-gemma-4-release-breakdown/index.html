How AI Agents Actually Work

From prompt to autonomous action — the architecture, tools, memory, and planning loops that make agents tick.

TL;DR

An AI agent is a language model in a while-loop with tools, memory, and planning. It reasons, acts, observes, repeats. The difference from a chatbot is scaffolding, not intelligence. Master these 7 building blocks and you can build, evaluate, or debug any agent system.

01 Agent ≠ Chatbot

A chatbot takes input and produces output. Once. An agent runs in a while loop. It receives a goal, picks an action, executes it, reads the result, and decides what to do next — autonomously, without you typing another word.

The model inside might be identical. The difference is scaffolding: tools it can call, memory it can access, and an orchestration loop that keeps it running until the task is done or a limit is hit.

Same model + different architecture = completely different capability. A single API call is a chatbot. A loop around that API call is an agent.

02 The ReAct Loop

Coined by Google Research in 2022, ReAct = Reasoning + Acting. The model alternates between thinking out loud (chain of thought) and taking actions (tool calls). Each action result feeds back into the next reasoning step.

// The agent loop in pseudocode while (task_not_complete) { thought = model.reason(goal, context, history) action = model.select_tool(thought) result = execute(action) history.append(thought, action, result) if (model.should_stop(result)) break }

That's it. Every agent framework — LangChain, CrewAI, Claude's tool use, AutoGPT — is some version of this loop with different levels of sophistication around error handling, planning, and memory management.

03 Tools: Giving the Model Hands

Without tools, a model can only generate text. Tools connect it to the real world. A "tool" is a function defined by three things: a name, a description, and a parameter schema.

The model reads the description, decides when to call the tool, and formats parameters per the schema. The orchestrator executes the actual function and passes the result back. The model never runs code directly — it just generates the request.

// Tool definition example { name: "web_search", description: "Search Google for current info. Returns top 5 results with titles + snippets. Best for factual queries about events or products.", parameters: { query: "string (2-100 chars, natural language)", max_results: "integer (1-10, default 5)" } }

Tool descriptions are 80% of tool reliability. The model selects tools based on descriptions, not code. A vague description = wrong tool selection = failed task.

04 Memory Architecture

Three types of memory, each solving a different problem:

Short-term — the conversation context window itself. Every message, tool result, and reasoning trace lives here. Finite and expensive. At 200K tokens, a complex workflow can fill it in minutes.

Long-term — vector database (Pinecone, Chroma, pgvector). Before each step, the agent queries for relevant past interactions. This is RAG applied to the agent's own history. The trap: irrelevant retrievals are worse than no memory at all.

Episodic — stored experiences, not facts. "Last time I tried scraping that site without delays, I got rate-limited. Use 2s between requests." The most underexplored memory type, and potentially the most powerful.

// Memory retrieval before each step relevant_memories = vector_db.query( embedding(current_context), top_k=5, threshold=0.82 // ← this threshold matters ) context = current_msgs + relevant_memories

05 Planning: Task Decomposition

Single-step tasks are easy. Multi-step planning is the hard problem. "Set up CI/CD for this repo" requires decomposing into dozens of subtasks with dependencies, ordering, and fallback paths.

The practical approach: hierarchical lazy planning. Generate a rough 5-8 step plan upfront, then decompose each step into specific actions only when you reach it. Don't over-plan — the information you need for step 7 probably doesn't exist yet at step 1.

The #1 failure mode of agent planning isn't bad plans — it's plan drift. After 20+ steps, the agent loses sight of the original goal. Build in periodic reflection checkpoints.

06 Error Handling in Production

Demo agents break on the first error. Production agents expect errors and handle them at three levels:

Layer 1: Retry — Exponential backoff (1s → 2s → 4s → 8s). Handles most transient failures: timeouts, rate limits, temporary outages.

Layer 2: Fallback — If the primary search API fails, use an alternative. If JSON parsing fails, extract from raw text. If one approach fails, try another.

Layer 3: Graceful degradation — Report what was accomplished, what failed, and why. A partial result with a clear explanation beats silent failure.

And the meta-layer: cost circuit breakers. A runaway loop can burn hundreds of dollars in minutes. Set per-task budgets and automatic shutoffs.

07 The Full Architecture

Wire it all together:

// Production agent architecture ┌─────────────────────────────────┐ │ ORCHESTRATOR │ │ Think → Act → Observe → Loop │ │ │ │ Tools: search, code, API │ │ Memory: short + long + episodic│ │ Planner: hierarchical lazy │ │ Error: retry → fallback │ │ Budget: cost circuit break │ ├─────────────────────────────────┤ │ HUMAN-IN-THE-LOOP CHECKPOINTS │ │ → external comms, prod writes │ │ → spending > threshold │ └─────────────────────────────────┘

Every production agent — Claude Code, Devin, Copilot Workspace — follows this pattern. The differences are in implementation details. But the bones are the same.

Full autonomy isn't the goal. Maximum useful autonomy is. Handle routine tasks independently, escalate high-stakes decisions to humans. That line is the real art of agent engineering.

Key Takeaways

  • An agent is a model in a loop. Same model + orchestration scaffolding = completely different capability.
  • ReAct (Reason + Act) is the universal pattern — think, act, observe, repeat.
  • Tool descriptions determine 80% of reliability. Invest in clear, specific descriptions.
  • Three memory layers: short-term (context window), long-term (vector DB), episodic (learned patterns).
  • Hierarchical lazy planning beats exhaustive upfront planning every time.
  • Production needs three error layers: retry, fallback, graceful degradation — plus cost circuit breakers.
  • The goal: maximum useful autonomy with human oversight at critical junctures.
