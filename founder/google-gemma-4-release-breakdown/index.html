How I Went From "Just Use Claude for Everything" to a 40/60 Local-API Split in One Week

April 2026 8 min read Symond — Founder
TL;DR

I was wrong about local models. Apache 2.0 licensing, not benchmarks, tipped the decision. Two weeks of blind testing showed zero quality difference on volume tasks. My local-API split moved from 20/80 to 40/60 in two weeks.

Section 01

The Shift

I'm going to be honest about the timeline here because I think it matters. Six months ago, I was the person confidently telling other founders to "just use Claude for everything." The quality gap between open-source models and frontier APIs was wide enough that the setup overhead of running local models wasn't worth it. I genuinely believed this.

I was wrong. Not completely wrong — the quality gap on complex reasoning is still real and still matters. But I was wrong about the gap on volume work, and I was wrong about how fast it would close.

Gemma 4's release this week forced me to actually test my assumptions instead of repeating them. Here's what happened.

The Moment

I set up Ollama on my M3 Max. Pulled Gemma 4. Took about 5 minutes. Then I ran a blind test: same volume tasks, randomized outputs. My accuracy at distinguishing them was barely above chance. That's when my thinking shifted.

Section 02

The Blind Test

Over a week of testing, I gave the same volume tasks — content generation, data processing, code scaffolding — to both local Gemma 4 and Claude API, randomized the outputs, and tried to tell them apart.

The quality gap on these task types has effectively closed. If quality is equivalent on volume tasks, then the only variables left are cost, latency, licensing, and control. And on all four of those variables, local wins.

Cost

$0 per token after hardware, versus $3-15/1M tokens for Claude. My hardware investment (already owned M3 Max) amortizes to zero within weeks at my token volume.

Latency

Local inference on M-series silicon is fast enough for all my batch processing needs. Not faster than API for single queries, but no network overhead and no rate limits.

Licensing

Apache 2.0 means I deploy without thinking about it. This matters more than it sounds. Every time I considered building a product feature on Llama, the 700M user cap created a conversation with myself about "what if we scale." With Gemma, that conversation doesn't happen. I just ship.

Control

The model runs on my hardware. No API outages, no pricing changes, no deprecation notices. The model I'm using today will work tomorrow because I control the infrastructure.

Section 03

The Routing Strategy

So I started routing. Volume tasks go to local Gemma 4. Complex reasoning goes to Claude. The split was 20/80 favoring Claude two weeks ago. It's 40/60 now. And every week I find another task type where local quality matches API quality, and the split shifts a little more.

I want to be clear about what this isn't: it's not a "Claude is dead" take. Claude is still clearly better at multi-step reasoning, strategic analysis, and nuanced creative work. I use it daily for those tasks and the quality difference is measurable.

The right answer is "both, routed intelligently." Not "all local" and not "all API."

Section 04

The Lesson

The lesson I keep coming back to: test your assumptions. I repeated "just use Claude" for six months without testing whether it was still true for all task types. It wasn't. The landscape moves fast enough that opinions formed in October are outdated by April.

If you're a founder running everything through paid APIs, I'm not saying you're wrong. I'm saying run the numbers. Route a week of volume tasks to local Gemma 4 and see if the quality difference matters for those specific tasks. It might not. And if it doesn't, the cost savings are immediate.

Key Takeaways
  1. 1 Apache 2.0 licensing is what actually tipped the decision. Not benchmarks, not cost savings, not performance — licensing. Being able to ship Gemma 4 in production without a lawyer, without a license review, without worrying about Llama's 700M user cap removed the last friction point that was keeping me on API-only.
  2. 2 The quality gap on volume tasks is functionally zero. Content generation, data processing, code scaffolding — on these task types, I genuinely cannot tell the difference between local Gemma 4 output and Claude API output. I tested blind for a week. The results were humbling.
  3. 3 Complex reasoning still clearly favors Claude. Multi-step reasoning, strategic analysis, nuanced creative work — Claude is measurably better on these tasks. This isn't a "ditch your API" story. It's a routing story.
  4. 4 My split moved from 20/80 to 40/60 in two weeks. And it shifts every week as I find more tasks where local quality matches API quality. The trajectory suggests 50/50 is coming, possibly within the next month.
  5. 5 The cost savings are real and trackable. Every task routed to local Gemma 4 is a task I'm not paying API tokens for. Over two weeks of tracking, the savings were meaningful enough to change my approach permanently.
